About the Project

What is OPTA?

OPTA stands for Onpremise Test Automation — it is a Jira project used to track automation work for Spinifex Solutions (Easy Reporter, Easy Clone, Easy Documents, and more).

Each OPTA-XX ticket in Jira represents a scenario to be automated. Linked to it is an OPSH-XXXX ticket (a TEST-type ticket) — this is where the actual test cases and Zephyr steps live.

Our automation goal: validate that each product behaves correctly across multiple SAP environments (MT and VT systems) before every release — and to support the manual execution of regression test cases, especially those run every patch.

Our Automation Tool

Introducing Playwright

Playwright is an open-source browser automation framework developed by Microsoft. It allows us to write code that controls a real browser — clicking buttons, filling forms, and verifying results — exactly as a human tester would, but automatically and at speed.

We write our test scripts in TypeScript (a typed version of JavaScript), which gives us better code quality, autocomplete, and error checking in VS Code.

Why Playwright for SAP WebGUI?

  • Supports XPath selectors — essential for navigating SAP’s complex HTML structure
  • Built-in auto-wait — handles SAP’s slow load times without brittle sleep calls
  • Runs a real Chromium browser, so what it sees matches what a tester sees
  • Native VS Code Test Explorer integration for running and debugging tests
  • Produces detailed screenshots and traces on failure for easy debugging

Phase 1 · Before AI

The Manual Approach

Before AI tooling was introduced, every test file was built by hand — a slow, technical, and error-prone process.

① Open Inspect Element

Open the browser's developer tools on the SAP WebGUI screen, then right-click on any element to inspect the HTML structure.

② Hunt for the Right XPath

The SAP DOM is deeply nested with generated IDs. The automation team had to trace up the DOM tree to build a reliable XPath that would still work after refreshes.

③ Write the Spec File Manually

Copy the XPath into a .spec.ts file, write each test step by hand, then run and debug until the test passes.

The bottleneck: A single test scenario with 8–10 steps could take half a day or more to automate — most of that time spent debugging broken locators rather than writing test logic.

Phase 1 · Debugging

The Debugging Cycle

Even after writing the test, getting it to actually run reliably against SAP's dynamic UI required a lengthy debugging process.

✗

Run the test → it fails

SAP loads slowly; elements sometimes don't exist yet when Playwright tries to click them.

⏱

Add waits and retries

Insert waitForTimeout calls between steps and increase actionTimeout to tolerate SAP's response time.

🔁

XPath resolves multiple elements

SAP renders similar labels in multiple places — the locator must be made more specific, often using positional indices like [1] or [2].

✓

Stable, passing test

After multiple iterations the test runs consistently — but the effort was disproportionate to the actual business logic being validated.

Key Challenges Faced

  • SAP reuses the same HTML structure for different controls — generic locators break easily
  • Dynamic element IDs generated by SAP change between sessions
  • Drag-and-drop actions in SAP tree views require precise coordinate calculations
  • Modal dialogs and popups appear unpredictably, blocking the automation flow
  • Every new test scenario meant starting the locator-hunting process from scratch

Phase 2 · AI Spec Generator

How the GitHub Copilot Skill Generates the Spec

1

Trigger the Spec Generator

The primary way is via the terminal command npm run generate:spec -- OPTA-91 — this is preferred as it avoids consuming GitHub Copilot tokens. Alternatively, the registered GitHub Copilot agent skill can be invoked by typing "generate spec for OPTA-91" in the Copilot Chat panel, but this uses more AI token budget and is better suited for one-off or exploratory requests.

2

Script resolves the linked Zephyr test

The generator queries Jira to find the linked OPSH-XXXX test ticket, then calls the Zephyr API to fetch every step (Action, Test Data, Expected Result).

3

AI pattern matching converts steps to code

Each plain-English step is matched against a library of known SAP patterns — navigation, dropdowns, tree drilling, drag-and-drop — and mapped to existing page object methods.

4

Spec file is written to disk

A tests/[NN]-OPTA-91.spec.ts file is created. Every test.step() block carries a [Zephyr Step N] comment. Steps the AI could not map get a /* TODO */ placeholder.

5

Fix TODOs & errors using “Record at cursor”

If a generated method call fails at runtime or a /* TODO */ needs a real locator, place the cursor at that line in VS Code and use Playwright → Record at cursor — the browser opens, you interact with SAP, and the exact locator is recorded and inserted automatically.

// generate-spec.ts – simplified flow const steps = await fetchZephyrSteps('OPSH-123'); for (const step of steps) { // Match navigation steps const tcode = detectTCode(step.action); // → tcodePage.accessEasyReporter() // Match known OPTA product names const product = detectProduct(step.action); // → "Go to Easy Reporter" → accessEasyReporter() // Match dropdown / button / drag patterns const action = detectGenericAction(step.action); // → unrecognised step → /* TODO */ } writeSpecFile(outputPath, specContent);

When the AI can’t map a step

  • A /* TODO */ comment is left as a placeholder
  • Use Record at cursor in VS Code to capture the real interaction
  • Same approach for any method call that throws an error at runtime

Deep Dive · Real Example

OPTA-91: From Zephyr to Running Test

Scenario: Easy Reporter — Calculation Fields Dropdown Validation — verifying that different calculation timing options can be assigned to calculation fields in a report.

What the Zephyr Steps Say (plain English)

  1. Log on to SAP. Navigate to Easy Reporter using transaction code /n/spin/er
  2. Click "Create a new Report"
  3. Go to Tab 2 (Report Columns) and add: Employee No., Calculation Fields 001–005, Date of Birth, Age
  4. For each Calculation Field, expand its Field Settings and set the Calculation Timing dropdown to a specific value
  5. Verify the expected result in the dropdown after each selection

What the Generator Produces

A test.step() block for each Zephyr step, with:

  • A call to tcodePage.accessEasyReporter() for navigation
  • Role-based locators for standard buttons (getByRole)
  • XPath patterns for tree nodes and dropdown fields
  • Comments referencing [Zephyr Step N] for traceability
test('OPTA-91: ER - Calculation Fields Dropdown', async ({ tcodePage, basePage, editReport, page }) => { await test.step('[Zephyr Step 2] Navigate to Easy Reporter', async () => { await tcodePage.accessEasyReporter(); await page.waitForTimeout(1500); }); await test.step('[Zephyr Step 3] Create a new Report', async () => { await page.getByRole('button', { name: 'Create a new Report' }).click(); await expect(page).toHaveTitle(/Create a new Report/); }); await test.step('[Zephyr Step 4] Add data fields', async () => { await editReport.reportColumns(); await basePage.drillDownReport([ 'Master Data', 'Employee Data' ]); await editReport.multiSelectAndDrag( ['Employee No.'], "//span[text()='Additional Report Columns']" ); }); await test.step('[Zephyr Step 5] Set calculation timing', async () => { await basePage.expandNode('Calculation Field 005').click(); await basePage.dropDown( '(02) Default (Calc before Filters)', 2, 2 ); }); });
Architecture · Page Object Model

Reusable Building Blocks

The framework uses the Page Object Model (POM) pattern — all interaction logic is abstracted into reusable classes, so test files stay readable and changes in the UI only need fixing in one place.

BasePage

  • drillDownReport() — navigate tree menus with auto-scroll
  • expandNode() — expand/collapse SAP tree nodes
  • dropDown() — select from SAP dropdown fields
  • selectTextButton() — click any element by its label

TcodePage

  • accessEasyReporter()
  • accessEasyClone()
  • accessEasyGoLive()
  • …one method per OPTA product — all use SAP's command field (#ToolbarOkCode)

Product Pages

  • EditReportPage — report column management, multi-drag
  • ERPage — Easy Reporter-specific actions
  • ECPage — Easy Clone flows
  • AuthPage — SAP login, session management

The question is: how easy is it to update when something changes? When a UI change happens in SAP (e.g., a label changes or a button moves), we only update one method in one file — not every test that uses that element. This is what makes Web GUI solutions significantly easier to maintain than deep backend tests.

Step by Step · OPTA-91 Full Walkthrough

How One Scenario Gets Fully Automated

1

Pick the Jira ticket

The team identifies OPTA-91 as a scenario to automate. The ticket is linked to a Zephyr test (OPSH-XXXX) with written test steps.

2

Run the spec generator

npm run generate:spec -- OPTA-91 — the tool calls the Zephyr API, reads each step, and scaffolds a [11]-OPTA-91.spec.ts file.

3

Review, fix TODOs & record missing locators

The automation team reviews the generated file. Any /* TODO */ placeholders or failed method calls are resolved using Playwright → Record at cursor in VS Code — interact with SAP live and the locator is captured automatically.

4

Pull changes & run via VS Code Test Explorer

Pull the latest changes from Bitbucket, then use the Test Explorer panel in VS Code to select and run the scenario. Watch the browser execute the steps live and fix any issues.

5

Commit & share via Bitbucket

Commit the finalised spec file to Bitbucket. Other testers pull the changes and run the test through their own VS Code Test Explorer. Jira is automatically updated with test results. Automated pipeline execution is still being researched by the web dev team.

6

Generate the CR / Automation Documentation

After the test passes, an Automation Specification Document (ASD) is generated — capturing the test scenario, methods used, page objects involved, Zephyr step mapping, and environment details. This CR doc is automatically published to Google Docs and linked back to the Jira ticket for full traceability.

Execution · How to Run

Running Tests via VS Code Test Explorer

The Test Explorer panel in VS Code is the main interface testers use to view, run, and debug automation tests — no terminal commands required.

1

Pull latest changes from Bitbucket

Use VS Code's built-in Source Control or any Git client to pull the latest spec files committed by the automation team.

2

Open the Test Explorer panel

Click the flask / beaker icon in the VS Code Activity Bar on the left side. This opens the Test Explorer panel showing all discovered Playwright tests.

3

Browse and select a test

Tests are listed by file and scenario name (e.g., OPTA-91: ER - Calculation Fields Dropdown Validation). Expand to see individual test steps.

4

Click ▶ Run

Press the run button next to the test. A real Chromium browser window launches and executes every step automatically — log into SAP, navigate, interact, verify.

5

View results inside VS Code

Passed steps show a ✔ green tick, failed steps show a ✗ red cross with the error message and a screenshot. Use the Debug button to step through a failing test interactively.

What You Can Do in Test Explorer

  • Run all tests — execute the full regression suite at once
  • Run a single test — target one scenario for quick validation
  • Debug a test — pause execution at any step and inspect the browser state
  • Filter by status — show only failed or passing tests
  • Re-run failed tests — retry only the tests that previously failed

After the Run

  • Results appear inline in Test Explorer immediately
  • Screenshots are saved to the test-results/ folder
  • Jira is automatically updated with the test status via the built-in reporter
  • Allure report can be generated locally for a detailed visual breakdown

The Question · Prioritisation

Web GUI vs. Core Backend — Ease of Automation

Not everything can be automated in 3 months. Here's how we think about where to invest automation effort first.

Dimension Web GUI Solutions
(Easy Reporter, Clone, etc.)		 Core Backend
(RFC, SAP standard config)
UI accessibility Browser-accessible — Playwright works natively No browser UI — requires API / RFC calls or SAP GUI scripting
Locator stability SAP-rendered HTML — complex but consistent patterns exist No HTML — different tooling required entirely
Update cost Page Object pattern means one-place updates Any change may require rewriting integration layer
Existing tooling fit Playwright + Spec Generator already working New toolchain needed (SAP API mocking, RFC stubs)
Zephyr traceability Full — step-by-step mapping already in place Partial — steps exist but linking to automation is manual
Time to first passing test Hours (with generator) Days–Weeks

Recommendation: Prioritise Web GUI automation first — the toolchain is already proven, the ROI per scenario is highest, and failures are visible and reproducible. Core backend scenarios should be tackled where they can be validated through the Web GUI (e.g., verify a backend calculation by reading its result in Easy Reporter).

The Question · What to Automate First

Prioritisation Strategy

✅ Automate First (High Priority)

  • Regression-critical scenarios — scenarios that break most often after a release
  • High-frequency manual test cases — tests run every sprint
  • Well-documented Zephyr steps — generator can scaffold these almost fully
  • Cross-environment checks — same test runs on MT5, MT6, MT8 without code changes

⚠️ Automate Later (Medium Priority)

  • Scenarios with frequent UI changes — maintenance cost is higher
  • Complex multi-system flows requiring external data setup
  • Exploratory / edge-case tests — better done manually by experienced testers

❌ Don't Automate (Low ROI)

  • One-off sanity checks
  • Scenarios that require human judgement on visual layout
  • Tests for deprecated features being removed soon
